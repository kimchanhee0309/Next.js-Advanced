#### SSG + ISR 하이브리드 패턴(generateStaticParams)
**`[id]`처럼 경로 파라미터를 받는 동적 패이지라도, 내용이 거의 변하지 않는다면 빌드 타임에 미리 정적 페이지로 만들어 `Full Route Cache`에 캐싱해둘 수 있음**

<br>

**언제 쓰면 좋을까?**
* 페이지 주소는 `id`마다 달라지지만(동적), 한 번 정해진 내용은 거의 바뀌지 않는 경우
* 예) 포켓몬 도감: `/pokemon/25`(피카츄)의 상세 정보는 한 번 정해지면 바뀔 일이 거의 없음
* 이런 페이지를 매 요청마다 새로 그리는 건 낭비 **→ 미리 만들어두면 응답이 빠르고 서버 부담도 줄어듦**

<br>

**어떻게 동작?**
`generateStaticParams`가 반환한 파라미터 목록만큼 빌드 타임에 정적 페이지를 생성함
```typescript
// api/[id]/page.jsx
export function generateStaticParams() {
  return [{ id: "1" }, { id: "2" }, { id: "3" }];
}
```
* **목록에 있는 id(1, 2, 3)**
  * 빌드 타임에 미리 생성해서 `Full Route Cache`에 저장. 첫 요청부터 바로 캐시에서 응답
* **목록에 없는 id(예: 4)**
  * 기본적으로 **첫 요청이 들어온 순간 생성**되고, 그 결과가 `Full Route Cache`에 저장됨. 이후 같은 요청은 캐시에서 응답
    * 단, 이건 `dynamicParams` 옵션이 기본값(`true`)일 때. `false`로 두면 목록에 없는 id는 생성하지 않고 404 처리
    * 여기서 "목록 밖 id 처리"는 매 요청마다 새로 그리는 진짜 dynamic rendering이 아니라, **요청 시점에 한 번 만들고 캐시에 굳히는 방식**(on-demand 정적 생성).
    * 페이지가 `cookies()`, `headers()` 같은 동적 API를 쓰면 그때는 캐시되지 않고 매 요청 렌더링됨
   
<br>

**주의: fetch 캐시 옵션**
데이터를 `fetch`로 가져올 때 `cache` 옵션의 동작을 이해해야 함

<br>

**`fetch`의 기본값 = `auto no cache`**
기본값은 단순한 "캐시 안 함"이 아니라 상황에 따라 갈리는 동작
* **개발 모드**: 매 요청마다 원격 서버에서 새로 가져옴(캐시 사용 x)
* **`next build` 시점**: 라우트가 정적 프리렌더링되므로 **한 번만 가져와 정적 페이지에 구워 넣음**
* **Request-time API**(`cookies()`, `headers()` 등)가 라우트에서 감지되면: 매 요청마다 가져옴(dynamic)

즉, **데이터 캐시(Data Cache)에는 저장하지 않지만, 빌드 타임 정적 프리렌더링은 기본 값만으로도 동작함**

<br>

**옵션별 정리**
|옵션|동작|`generateStaticParams`와의 관계|
|---|---|---|
|`auto no cache`(기본)|빌드 시 1회 fetch 후 정적 프리렌더링|별도 설정 없이 정적 생성됨|
|`force-cache`|결과를 Data Cache에 영속 저장(요청 간 재사용)|데이터까지 캐시하고 싶을 때 명시|
|`no-store`|매 요청마다 새로 가져옴 -> 라우트가 dynamic 처리|정적 생성이 깨짐(피해야 함)|

* 정리:
  * 빌드 타임 정적 생성만 원하면 **기본값으로 충분함**
  * `force-cache`는 데이터를 캐시에 영속적으로 굳혀 요청 간 재사용하고 싶을 때 명시함
  * `no-store`는 라우트를 dynamic으로 만들어 정적 생성을 깨뜨리므로 이 패턴에선 피함
 
<br>

**한눈에 정리**
|구분|생성 시점|캐시 동작|
|---|---|---|
|목록에 있는 id(1,2,3)|빌드 타임|`Full Route Cache`에 저장|
|목록에 없는 id(`dynamicParams:true`)|첫 요청 시(on-demand)|첫 생성 후 `Full Route Cache`에 저장|
|목록에 없는 id(`dynamicParams:false`)|생성 안 함|404 반환|

* `[id]`처럼 **경로 파라미터를 받는 동적 페이지**라도, 내용이 거의 변하지 않는다면 **빌드 타임에 미리 정적 페이지로 만들어 `Full Route Cache`에 캐싱**해둘 수 있음

#### SSG + CSR 하이브리드 패턴(Tanstack Query with Next.js 패턴: prefetching, dehydrate, and hydrate)

`SEO`와 `UX 향상`의 효과가 있음
* **`SEO`**: 일반적인 SSSR 페이지와 마찬가지로 100% 프리렌더링 효과를 가짐
* **`UX 향상`**: 캐시(queryData)를 가지기 때문에 다른 페이지에서 다시 해당 페이지로 돌아왔을 때 로딩없이 바로 화면을 볼 수 있을뿐만 아니라 queryFn의 백그라운드에서 실행되며 최신 데이터를 금방 확인할 수 있음
```typescript
// app/posts/page.jsx
import {
  dehydrate,
  HydrationBoundary,
  QueryClient,
} from '@tanstack/react-query'
import Posts from './posts'

export default async function PostsPage() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    // Neat! Serialization is now as easy as passing props.
    // HydrationBoundary is a Client Component, so hydration will happen there.
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}
```
